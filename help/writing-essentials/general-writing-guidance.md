# 外部寄稿者向けのオーサリングスタイルガイドライン{#guidelines}

このページでは、外部の作成者がコンテンツを作成したり、Experience League上の既存のコンテンツを更新したりする際の編集上のガイドラインを示します。 開始する前に、次の事項を確認します。

* 詳しくは、 [Markdown](markdown.md) オーサリング
* 記事のつづりや文法を確認してください
* わかりやすいトーン、一貫性のあるプレゼンテーション、シンプルな文を使用して、機械翻訳を改善します
* フォロー [ベストプラクティス](#writing-tips) このページの編集基準

## スタイルのガイドライン{#style-guidelines}

ドキュメントを記述する際は、次の点に留意してください。

* **簡潔に書く**：冗長な表現を避けます。文は短く簡潔にします。記事の焦点を絞ります。メモの数は最小限にします。
* **対象と目的を絞る**：書き始める前に、読み手は誰で、何のタスクを実行しようとしているかを明確にします。読み手がそのタスクを実行するときに役立つような記事を書きます。
* **例を示す**：概念を説明する例を示します。
* **コンテンツを整理する**：いくつかのセクションを設け、手順をいくつかの扱いやすいグループに分けるようにします。画面を示した方がわかりやすいときは、スクリーンショットを追加します。

## 技術的な記述のベストプラクティス{#writing-tips}

特にソフトウェアドキュメントに関する技術的な記述は、特に専門業界です。 最も多作な小説家でさえ、技術的な書き込みを試みるときに驚かされる。素材が複雑で技術的なものだからではなく、複雑で技術的な情報を作るのは容易ではないからだ。 _単純_. 成功させるには、構造や構文のエラーなしに、コンテンツの構造的に一貫性があり、スキャン可能で、再利用可能で、公開パイプラインを流れる必要があります。

以下の節では、新しいライターが注意する必要がある一般的な問題について説明します。

### 見出しをテキストで区切らない（見出しを二重にする）{#double-headings}

2 つの見出しを持ち、それらを区切らない場合は、見つからないテキストを追加します（2 番目のトピック見出しが表示されます）。 または、見出しを削除することもできます。 2 つ目はおそらく不要です

例： _概要_ ここには何の目的もありません。

![見出し（二重）](assets/headings-double.png)

* また、2 番目の見出しが _概要_&#x200B;必要ないかもしれません。 H1 と最初の段落は、記事のトピックに関する概念的概要として機能します。

* 同様に、SEO の目的では、 _概要_ および _はじめに_ 自分では役に立たない。 導入する製品または機能に名前を付けます。 ( 例： _フォールアウトレポートの概要_)

### 相互参照の見出しに一貫性がありません{#maps}

用途 _詳細情報_ 相互参照リスト（またはマップ）の見出し。 例：

![相互参照リスト](assets/headings-more-info.png)

**相互参照リストのガイダンス**

* クロスリファレンスに箇条書きリストを使用
* ガイドまたはページ名の正式名には斜体を使用します（リンクテキストを使用しない場合）
* 見出し（または見出し）を中断しない
* 見出しでの数値の使用

### TOC エントリ、パンくずリスト、ページ名が一致しません{#toc}

TOC（目次）ファイルを手動で管理するので、これらの不一致は簡単なミスです。 TOC エントリがページ名 (H1) と一致することを確認します。 また、パンくずリストと密接に一致することを確認します。

**目次とリストに関するガイダンス**

* 目次のエントリの短縮が必要になる場合がありますが、このエントリは、ページ名とパンくずリストに明確に関連付けられている必要があります。
* パンくずリストはタイトルメタデータから取り込まれるので、異なる値を指定できます（SEO 用）。

### 斜体の代わりに引用符を使用する{#quotes}

単語やフレーズを引用符で囲むのに抵抗するのは困難です。 ただし、引用符は音声の引用を目的としており、製品ドキュメントではほとんど使用されていません。

**引用符に関するガイダンス**

* 通常、斜体の方が引用符よりも適しています（エラーメッセージ、一意または外部の単語など）。
* インターフェイス要素には、太字と UICONTROL を使用します。

### 手順{#steps}

プロシージャの記述 ( _タスク_ コンテンツタイプ ) は、私たちが生まれた才能ではありません。 読み取り可能で明確な手順を構築する必要があります。

**手順のガイダンス**

* 手順は、一連の手順です。 一歩は短く、数えられている。 _1 文の_ コマンドを使用します。
* 各手順を動詞または _宛先_ 不定詞（読者を目標に向ける、のように） _サインイン状態を維持するには、**サインイン状態を維持**_) をクリックします。 手順全体で特定の目標が設定されている場合は、アクションの前に目標を指定します。
* ステップに関する情報がある場合 ( _手順情報_&#x200B;をクリックし、ステップの後（ステップでインデント）またはアセットの後（スクリーンショット、ビデオまたはインターフェイス説明のリスト）に追加します。
* ステップに 2 つのアクション ( _これを選択し、次に_)、単一の短い文として書きます。
* タスクを約 7～10 ステップに制限します。 1 つのタスクで 10 個を超えるステップを作成する場合は、2 つのタスクに分割する必要が生じる場合があります。 ここで最善の判断をしなさい。
* 製品ドキュメントでは、見出しを手順として使用しないでください。 （チュートリアルの場合は以下の例外）
* 複数ページのチュートリアルでは、手順としての見出しを使用できます。 ただし、番号を付けないでください。 代わりに、詳細を説明します。 _手順 1:_, _手順 2:_&#x200B;など。

**手順の例**

次に、Adobeでのログインに関する適切な構造の手順を示します。

Adobe:

1. オン `Adobe.com`を選択します。 **Experience Cloud**.
1. 選択 **ログイン**.
1. 選択 **個人用アカウント**.
1. サインイン状態を維持するには、 **サインイン状態を維持**.
1. 名前とパスワードを入力します。
1. 選択 **ログイン**.

### 並列リスト{#lists}

リストに対して並列構造を使用すると、読み取りとスキャンが容易になります。 リストには、目次（目次）、箇条書き（順不同）、番号付きリストが含まれます。

並列エントリを含む TOC の例：

![並列目次](assets/parallel-toc.png)

前述の TOC は、次の理由で好例です。

* 概念的な親エントリは、名詞または名詞句です
* 手順（タスク）は、有効な動詞（逆引きではない）です
* すべてのエントリでは文の大文字が使用されます

## タイトルと説明のメタデータ{#metadata}

_タイトル_ および _説明_ メタデータは、SEO、コンテンツ検出、Experience League上のコンテンツ品質スコアにとって重要です。

タイトルと説明の例を次に示します。

**コンセプト記事の説明**

* _Adobe Analyticsのセグメントについて説明します。 ワークスペースでのセグメント化パネルの設定に関するヘルプを表示します。_
* _Adobe Analyticsのページビュー数レポートでのセグメントの使用に関するヘルプを見つけます。_

**手順/タスクに関する記事の説明**

* _Adobe Analyticsでセグメントを作成する方法を説明します。_
* _Adobe Analyticsでセグメントを作成します。 作成したセグメントに基づいてレポートを選択、設定、実行する方法について説明します。_

使用する記事のサイズと範囲に応じて異なります。

**概念記事のタイトル**

* _ページビュー数レポートのセグメント_

**手順/タスク記事のタイトル**

* _ページビュー数レポート用のセグメントの作成_

（パイプ名と製品名はタイトルに自動的に追加されます）。

## 代替テキスト{#alt-text}

アセット（画像）に意味のある代替テキストを追加します。 次の条件に一致する代替テキストを検討します。

* お客様が達成できる目標（タスク名またはコンセプト名）
* 表示している機能またはページ
* 表示するアイコン名

Googleは SEO 結果で alt-text を考慮します。

## ローカライゼーション — DNL および UICONTROL{#localization}

製品がローカライズされているか、ExL が使用する言語かを気にする必要はありません。 ただし、必要に応じて次の 2 つ（必須）の Markdown タグを適用することで、ローカライゼーションの品質を向上させることができます。

* `DNL`

   DNL は次の意味を持ちます。 _ローカライズしない_. 商標を示すAdobe製品名に対してのみ使用し、すべて英語で記述する必要があります。

   構文の例： `[!DNL Adobe Campaign]` または `[!DNL Workfront]`

   DNL は、ファイル名や URL を対象としていません。

* `UICONTROL`

   UICONTROL は、インターフェイスコントロール（オプション、フィールド、タブ、ページ、オプションのグループ、UI の機能名など）を示します。

   構文の例： `Select **[!UICONTROL Project]**, then select **[!UICONTROL Save]**.`

>[!IMPORTANT]
>
>コンテンツのローカライズの前に、これらのタグを適用する必要があります。

### 製品名でのAdobeの使用{#product-names}

企業 ID の場合、通常は _Adobe_ （ガイドレベルでの製品の最初の参照） スペースに関する考慮事項に応じて、見出しにAdobeをドロップできますが、本文のコピーの最初の参照には完全な名前を含める必要があります。 特定の製品 ( 例： _Adobe Audition_ および _Adobe Premiere Pro_&#x200B;では、Adobeは法的な商標名の一部なので、あらゆる担保の中で最初または最も目立つ参照に関する商標を使用する必要があります。

## 最初の段落{#firstparas}

最初の段落では、トピックを定義し、記事を読む読者が学習する内容を説明する必要があります。

最初の段落の例（概念）:

_オーディエンスは、訪問者の集合（訪問者 ID のリスト）です。Adobeのオーディエンスサービスは、訪問者データからオーディエンスセグメントへの変換を管理します。 したがって、オーディエンスを作成および管理することは、セグメントを作成および使用することに似ています。また、オーディエンスセグメントをExperience Cloudと共有することもできます。_

最初の段落の例（タスク）:

_顧客属性ソース（CSV および FIN ファイル）を作成してデータをアップロードします。準備できたら、データソースをアクティブ化できます。データソースがアクティブになったら、属性データを Analytics と Target で共有します。_

### 最初の段落の SEO のヒント{#seo}

* 最初の段落に検索語句を含めます。
* 読者が使用する用語を使用します。
* 同義語を含め、必要に応じて、以前の用語の使用方法も含めます。 例えば、「Experience CloudID サービス (ECID)：旧称 _訪問者 ID_ また、MID、MCVID などの略語として、訪問者を識別する永続的な汎用 ID を提供します。」
* リンクに SEO 用語を含めます。
* 複雑なテーブルに必須の用語を配置しないでください。 複雑なテーブルでは、信頼性の高い検索結果は得られません。 画像内のテキストは検索ではありません。 キャプションが検索されます。

## 大文字/小文字{#capitalization}

* Adobeスタイルでは、すべてのタイトル、見出し、サブ見出しおよびページナビゲーション要素に対して、文スタイルの大文字/小文字を使用します。
* ブランド、ソリューション、サービスの名前など、最初の単語と固有名詞を除くすべての単語は小文字です。
* ツール、オプション、メニュー項目、ダイアログボックス、フィールドの製品名の大文字と小文字を一致させます。

## 目次{#using-toc}

この `TOC.md` は目次です。 各ガイドには 1 つずつ必要です。

**目次の編集上のガイダンス**

* 大文字：各エントリでは必ず文頭文字を使用します（略語は含まない）。 正式な製品名またはインターフェイス要素（ページ、タブ、フィールド、オプションなど）のみを大文字にします。 参照元の UI と一致させます。
* 動詞の形式と並列性：命令動詞を使用し、回避策を避けます。 目次はリストなので、ほとんどの場合、常にリストを並行に保つようにします。 回避できない例外もあります。 概念ページには、名詞と名詞句を使用します。 タスクには、動詞を使用します。

**構文ガイダンス**

* 目次のセクション見出し（親）をリンクにすることはできません。コンテンツを含むページはありません。 これには、次のようなアンカーを含める必要があります。 `{#processing-rules}`.
* TOC セクションの見出しには、正しい構文を使用する必要があります ( 例： `+ Processing rules {#processing-rules}`) や TOC 記事のリンク ( 例： `+ [Article name](article.md)`) をクリックします。
* TOC 記事のエントリは、記事のタイトルを短くしたバージョンにすることができます。 このドキュメントの概要、概念、タスクを記述する際の基準に従います。
* 同じファイルを複数回目次（または複数の目次）に追加しないでください。 そうすると奇妙な動作を引き起こします。
* リポジトリに複数のユーザーガイドが含まれている場合、ユーザーガイドディレクトリは、( `help` ディレクトリ。 各ユーザガイドディレクトリには TOC ファイルが必要です。 ネストするユーザガイドはありません。

## 太字と斜体{#bold}

* 手順内（および UICONTROL を使用）でクリックしたインターフェイス要素に対してのみ、太字テキストを使用します。
* 強調にイタリア語を使用したり、単語を使用しないと混乱を招く場合に使用します。 例えば、外国語の単語や、単語を表したり用語を定義したりする場合です。
